Abstractness versus Instability Diagram

The Abstractness versus Instability Diagram helps to detect which assemblies are potentially painful to maintain (i.e concrete and stable) and which assemblies are potentially useless (i.e abstract and instable).

  • Abstractness: If an assembly contains many abstract types (i.e interfaces and abstract classes) and few concrete types, it is considered as abstract.
  • Stability: An assembly is considered stable if its types are used by a lot of types of tier assemblies. In this conditions stable means painful to modify.

Code Quality from Now!

  • On a large existing code base, the NDepend set of default CQL rules will likely report thousands of flaws and imperfections.
  • It is not possible to stop development for several weeks to fix all flaws, and then resume development on a clean code base.
  • NDepend has support for the Code Quality from Now! strategy. The idea is to ensure that flaws are fixed only on code added and eventually, on code refactored as well.
  • The first thing to do is to capture the current state of the code base and set it as a Baseline for Comparison.
  • The second thing to do is to adapt and customize CQL Rules to the Code Quality from Now! strategy. Concretely you just need to append the CQL clause AND (WasAdded OR CodeWasChanged) to a CQL rules to get it checked only on new and refactored code.
  • For example, the following CQL rule reports only added and refactored too complex methods.
    // <Name>From now, all methods added or refactored must be not too complex </Name>
    WARN IF Count > 0 IN SELECT METHODS WHERE
           (WasAdded OR CodeWasChanged) AND CyclomaticComplexity > 20
        
  • Another example is to ensure that all new or refactored types respect a minimum of ratio of code coverage by tests.
    // <Name>From now, all types added or refactored should be 70% covered by tests </Name>
    WARN IF Count > 0 IN SELECT TYPES WHERE
           (WasAdded OR CodeWasChanged) AND PercentageCoverage < 70
